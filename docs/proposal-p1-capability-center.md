## 1. 归属裁定(§1 铁律 · 最关键,先钉死)

**能力中心 = 平台 / 壳管理面,归 `platform/`,不塞进任何 `apps/`。**

- **理由**:能力(MCP 连接、记忆、知识库、技能……)是**平台级、跨应用共享**的资源;管理它天然是平台职责。塞进某个 app 会让「能力」依附于一个可被关掉的业务应用 → 违反「apps=业务、platform=业务无关」。
- **先例**:与「应用管理页」(开/关/排序应用)同性质——都是壳管理面,已归 `platform/shell/`。
- **数据归属**:Connector 配置 / Skills / Scheduled 定义 = **平台集合**(非某 app 用 `<appId>_` 前缀认领);记忆(`LongTermMemory`)、知识库(`DocContext`)**已是平台能力**(capability.rs 已注册)。
- **反面明确**:assets 应用**不是**「重构成能力中心」——它退役、数据迁入平台能力(§5)。

## 2. AI-Native:能力中心怎么在 Agent 窗口里(不是又一个设置页)

用户核心痛点是「非 AI-Native、任何功能都应能在 Agent 窗口进行」。故能力中心**不是深藏设置里的表单**,而是:

- **既是画布视图**(`data-canvas`=能力中心,复用 P0 的右画布),**也在对话里可达**:「我连了哪些 connector?」「有哪些技能可用?」→ Agent 直接列举/解释(**读操作**)。
- **能力流入 Agent**:一个 connector 连上 → 它的工具进入工具循环 → Agent 在对话里就能调。这就是「管理能力」与「使用能力」在同一窗口闭环 = AI-Native。
- **★红线守界(§4-2)· 关键取舍**:**配置变更(加 connector / 填密钥 / 启用 / 定时任务)= 设置类,不可经对话改**;Agent 只**引导**去能力中心管理面。**读(列举/查看/解释)走对话,写(配置/密钥/启停)走管理面**。这条把「AI-Native」与「设置不可经对话改」两条不矛盾地并存。
- **★读路径 = 有界静态投影,不是「任意读进模型」(评审 [应改]A · 载重,拍板前必钉)**:「读走对话」**绝不等于**把连接器原始记录喂给模型。`mcp_list` 返回体含 **`url`(端点)/ `command` / `args`(本地 stdio 命令)**([mcp.rs:890-892](src-tauri/src/mcp.rs#L890))——**若图省事复用它喂模型,用户的 MCP 端点 + 本地命令路径会进(多为远程第三方的)BYO 模型上下文 = 基础设施拓扑泄露、违本地优先 §4-1**(注:auth/env 已是状态-only `configured/empty`、非值,mcp.rs:886;故严重度低于 route-A 的密钥泄露、但仍是结构缺口)。故**模型侧读路径只走一个静态最小投影**:`name + enabled/connected 状态 + 工具名/描述`(镜像模型经 `tool_descriptors` 已见的面,[ai.rs:432-435](src-tauri/src/ai.rs#L432)),**显式排除 url/command/args/auth/env/一切配置细节**。**管理面视图**(画布里给人看)可读全量 `mcp_list`,但**全量永不入模型**。此纪律与 D3 的「静态 QUERYABLE 硬底」同源(见 §7 新红线)。

```
┌──────────── Agent 窗口 ────────────┐
│ 对话流(左)          │ 画布(右)          │
│ 「有哪些能力?」→列举 │ 能力中心视图:       │
│ 「加个 MCP」→引导    │  · Connector(已连/工具)│
│ 工具调用可见          │  · 记忆 / 知识库(查删) │
│                      │  · Skills / …(绿地占位)│
└────────────────────────────────────┘
  读→对话 · 写(配置/密钥)→管理面(红线)
```

## 3. Connector-MCP 最薄先落(P1 第一刀 · 唯一「已建后端」)

**为何它先**:六项能力里,**只有 MCP 有成形后端**——`mcp.rs`(1669 行)+ 命令齐(`mcp_list/add/set_auth/set_env/remove/set_enabled/probe/confirm_resolve`)+ 确认专路(Untrusted 回灌)。**先用它把能力中心的范式跑通,是最省、最快的第一刀。**

- **现状**:连接器管理**埋在设置页深处**(`web/platform/shell/settings.js`)——功能在、但不是一等公民、不 AI-Native。
- **P1 做的**:把 Connector 从「设置页一段」**提为能力中心一等视图**(平台壳)。
- **★成本三分,别一把「零新后端」盖(评审 [应改]B 订正)**:
  1. **管理面视图**(画布,给人看:已连 connector + 状态 + 工具 + 配置)= 读全量 `mcp_list`、**纯前端渲染,零新后端** ✓。
  2. **模型「我能做什么/有哪些工具」** = **模型已免费持有**——`tool_descriptors` 已把 MCP 工具名 + server 名(嵌描述)+ schema 喂进工具表([ai.rs:432-435](src-tauri/src/ai.rs#L432)),**零新后端、零新暴露**,这问模型直接能答。
  3. **模型级 connector 状态叙述**(工具清单之外的 connected/error 等)= **唯一需要新面**:要么新增一枚投影能力(经 §2 [应改]A 的静态最小投影)、要么纯前端(模型不参与)。**二选一,拍板时钉**;不把它混进「零新后端」。
- **红线复用(不新造)**:密钥仍只进钥匙串(`mcp_set_auth` 已守、前端只见 configured/empty);MCP 结果 `Untrusted` 回灌专路(ai.rs 已守);启停即刻影响工具循环可用集。
- **[建议]C · 工具枚举永经 registry/契约、绝不 import app**:能力中心「列工具」= 读 `cap_list`(平台命令)/ registry,**平台读平台、无 platform→app import**。注意 registry 现含 `jobseek_market_value`([capability.rs:181](src-tauri/src/capability.rs#L181),route-B 打样债)——能力中心会把这枚业务名工具显示为平台能力(**非新 §1 破,是第52轮已记打样债**)。**app-tool 契约落地(工具从 Rust registry 迁到前端契约)时,「列工具」路径须同步更新**,否则契约迁移后能力中心列不全或被迫 import app。

## 5. assets 拆两半(§7.4 落地 · 归属在此定,数据迁移是 P2)

- **(a) 能力管理 UI** → `platform/`(即本方案的能力中心)。
- **(b) assets 的 notes / prompts 数据** → 迁入平台能力:notes → 记忆/知识库、prompts → Skills;**assets 应用退役(数据保留、可关)**,**非**「把 assets 重构成能力中心」。
- **P1 不动 assets 数据**——只把「家」(能力中心 + 记忆/知识库视图)建好;**数据迁移 + assets 退役 = P2**(与 jobseek 真化同期,见 proposal-agent-native §3-P2)。
- **★[建议]D · 退役有排序耦合、非一次性**:notes → 记忆/知识库(后端已建、可迁),但 **prompts → Skills 迁不走**——Skills 后端零基础(§4 绿地)。**故 assets 完整退役(P2)须等 Skills 后端落地**;P2 切勿假设「一次性退役」而卡壳:可先迁 notes、prompts 待 Skills 就绪再迁,或 assets 分步退役。P1 不动数据、此耦合只影响 P2 排期。

本地优先 · profile 硬隔离(能力中心**永不**触 profile)· D3 静态 QUERYABLE 硬底(勿改动态)· **配置/密钥/启停不可经对话改、Agent 只引导**(§2 关键取舍)· 密钥只进钥匙串 · MCP/外部内容 `Untrusted` 标注 · 破坏性(删记忆/删知识库文档)预览+确认+撤销走 guardrail · platform/apps 物理分离只靠契约 · 能力中心是平台壳、不 import 任何 app。

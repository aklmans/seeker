## 1. 目标与非目标

**目标**
1. **应用管理(时光序式)**:每个小应用可**开 / 关 / 排序**;关 = UI / 导航 / 卡片 / AI 工具全下架,**数据保留**;开即恢复。
2. **平台服务全共享**:AI 网关 / 数据层 / 钥匙串 / MCP / RAG / 记忆 / 护栏 / 受控出口 / 设计语言 / i18n,一份,应用只消费。
3. **新应用边际成本低**:一个 manifest + 一组页面模块 + (可选)卡片与提示 overlay,注册即上架。
4. **隐私升级(为健康类应用铺路)**:AI 可读范围 = **启用应用 ∩ manifest 默认可读 ∩ 用户 per-app 授权** 三层闸。

**非目标(本轮不做)**
- 不做第三方应用 / 插件市场——全部**第一方、编译进包**(沙箱化三方插件是远期,复用 show_widget 三道墙思路)。
- 不引前端框架、不换存储引擎、不动密钥 / 出口 / profile 红线。
- 不做跨应用数据联动总线(先各自独立;真实需求出现再设计)。

## 2. 概念模型(三层)

```
┌────────────────────────── 壳 shell(新)──────────────────────────┐
│ 启动器/应用网格 · 导航装配 · 应用管理页 · 全局设置 · Copilot chrome │
│ 首页聚合卡(各启用应用贡献摘要卡,时光序式)· 主题/i18n 装配         │
├──────────────────────── 小应用 apps/*(新)───────────────────────┤
│ jobseek(求职,现单体)│ assets(数据资产)│ ledger(记账)│ health… │
│ 每个 = manifest + pages + cards + prompts overlay + settings 段     │
├───────────────────────── 平台 platform(不动)─────────────────────┤
│ rt.*(db/ai/secret/capability/profile/memory/docs/mcp/render/web)   │
│ Rust 核:SQLite/钥匙串/AI 网关/MCP/RAG/受控出口/docx …             │
└─────────────────────────────────────────────────────────────────┘
```

**App manifest 草案**(TS 类型,`web/platform/shell/types.d.ts`):

```ts
interface AppManifest {
  id: string;                       // 'jobseek' | 'assets' | 'ledger' …(嵌入集合名/设置键,须 ^[a-z][a-z0-9]*$)
  name: { zh: string; en: string };
  icon: string;                     // 细线 SVG(合设计语言)
  blurb: { zh: string; en: string };// 应用网格一句话简介
  collections: string[];            // 声明拥有的数据集合(D1:白名单 = 各应用声明之并集)
  aiReadable: 'default-on' | 'default-off';  // D3:AI 可读默认档(健康类 default-off)
  nav: Array<{ page: string; title: {zh,en}; icon: string }>;  // 贡献的导航项
  pages: Record<string, (host: HTMLElement, ctx: ShellCtx) => void | (() => void)>; // 挂载器(返回卸载器)
  cards?: Record<string, CardSpec>;         // 贡献给 SEEKER_CARDS 的卡
  frameQuery?: (text: string, ctx) => string | null;  // 贡献的意图框定(启用才参与)
  homeCards?: (ctx) => HTMLElement[];       // 首页聚合卡
  settings?: (host, ctx) => void;           // 应用级设置段(全局设置仍归壳)
  seed?: (ctx) => Promise<void>;            // 首启示例数据(标「示例」角标,套现有种子守卫)
}
```

**ShellCtx** = `rt.*` + 壳服务(`tt/toast/openModal/guardrail/theme`)。**应用间互不 import**;唯一交汇点是壳与平台。

**生命周期**:全部应用**静态装载**(D4),启用状态存 settings KV(`apps.enabled`)。启用 → 注册 nav/cards/frameQuery/AI 集合;停用 → 全部注销 + 已挂载页面卸载。**关 ≠ 删**:数据保留;删数据是应用管理页里的独立动作,走 guardrail(预览 + 确认 + 可撤销)。

## 3. 关键决策点(D1–D7,各附推荐,等你拍板)

| # | 决策 | 选项 | 推荐及理由 |

## 4. 分阶段推进(每阶段可独立演示、过审再进下一阶段)

**阶段 0 · 方案对齐 + 基线更新**(本文档过审 → 修订 `app/CLAUDE.md` §1/§3/§5;`prototypes/` 修订由你定)

**阶段 1 · 壳与契约落地**(关键的一刀,行为零变化)
- `web/platform/shell/`:AppRegistry + manifest 类型 + 导航/设置/Copilot chrome 的装配权移交壳;
- jobseek manifest 以**适配器**包住现有单体函数(页面挂载器直接调既有 render 函数);
- **验收 = 现功能零回归**(9 页 + 双模式 + 设置全好用),外审。

**阶段 2 · 应用管理页**(时光序式)
- 应用网格:开 / 关 / 排序 / 简介 / 数据体量提示;「清除该应用数据」走 guardrail;
- 启用状态接通 nav / cards / frameQuery / AI 集合闸(D3 的运行时 QUERYABLE);
- **验收 = 关掉求职,壳、设置、Copilot(通用对话)仍完整可用**。

**阶段 3 · 求职模块化搬迁**(渐进,无固定截止)
- 单体里的求职页**逐页**迁入 `web/apps/jobseek/`(i18n 文案随迁),每页一 commit;
- 迁完删适配器;index.html 收缩为壳的骨架页。

**阶段 4 · 第二应用「数据资产管理」**(验证新增成本)
- 记忆 / 知识库 / 会话史 UI 化(平台能力现成)+ Prompt 库 / 笔记(新集合 `assets_prompts` / `assets_notes`);
- **验收 = "新增一个应用"只需 manifest + 页面模块,平台零改动**(此即平台化成立的证明)。

**阶段 5 · 后续应用逐个上**
- **记账**:订阅 / 采购(新集合)+ **token 用量**(唯一预见的平台层小改:AI 网关埋用量事件——非红线,过审即可);
- **项目管理**;**健康管理**(隐私分级示范:`default-off` + 首启明示)。

**建议的前置动作**:阶段 1 动工**前**先做一次 R1 macOS GUI 冒烟(拿到重构前的目测基线)——大规模 UI 重构叠加"从未真机目测"是双重风险,先固定基线再动刀。

## 5. 数据与导出兼容

- 导出 bundle 增记 `apps`(id + 版本 + 集合清单);导入遇**停用**应用的数据 → 照常导入(数据归属不随开关变);遇**未知**集合 → 保留 + 提示(前向兼容)。
- `messages`(会话史)/ `profile` / `settings` 仍全局归壳与平台,不属任何应用。
- 种子数据:per-app `seed()`,示例数据标「示例」角标(时光序同款),套现有种子守卫。

## 6. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 单体拆分回归(最大风险) | D5 适配器先行 + 逐页搬 + 每步 node/tsc/build + 每阶段外审;**前置 R1 冒烟固定基线** |
| 首包体积(全量静态装载) | 观测;超阈值再上懒加载(D4 留口),不预先优化 |
| i18n 文案随页搬迁散失 | 每页迁移 commit 里 zh/en 成对走,tsc + 人审 |
| 「Seeker」名字带求职色彩 | 产品命名是开放问题(§7),不阻塞技术推进 |
| AI 三层闸把现有对话弄哑 | jobseek 默认启用 + default-on,升级后行为不变;闸只对新增应用生效 |

## 7. 开放问题(不阻塞拍板,过程中定)

1. 产品名与定位(Seeker → ?);
2. 健康类应用默认 AI 策略细则(default-off 之外,是否连"启用后询问"都要);
3. 跨应用数据引用(如记账引用求职的 offer 薪资)——出现真实需求再设计,当前禁止互 import;
4. 应用管理页要不要"数据体量 + 最近使用"提示(时光序没有,但合我们"用户掌控"味);
5. `aiReadable` 目前是 per-app 单档(第5轮审查提示):将来某应用若含**敏感度不一的集合**(如项目管理的"公开笔记"与"薪资谈判记录"),或需 per-collection 粒度——届时再升,不预先设计。
